# PipelineAgent: Architecture & Execution Semantics

This document defines the core architectural decisions and execution semantics of PipelineAgent. All engine-level implementations and refactoring must strictly adhere to the following system contracts.

## 1. Execution Model

The engine is responsible for transforming the DAG blueprint generated by the Planner into actual computational tasks.

* **Scheduling Policy:**
    The engine adopts **Topological Sort + Greedy Scheduling**.
    The engine does not implement complex priority-based scheduling. As soon as a node's dependencies are resolved, it immediately enters the corresponding `Runtime` (such as `LOCAL_CPU`, `LOCAL_GPU`) semaphore waiting queue. Tasks within the same runtime queue are executed in FIFO (First-In-First-Out) order.

* **Variable Injection Rule:**
    Data transfer between nodes strictly uses **string interpolation** in the format `${node_id.output}`.
    The engine does **not** perform JSON `dump/load` or complex type inference at runtime. If a downstream Tool requires a `dict` or `list`, the Tool developer must use `pydantic` within the Tool to handle type conversion and validation. The engine only guarantees precise string replacement.

## 2. State Management & Idempotency

In distributed and asynchronous DAG execution, ensuring state correctness and avoiding duplicate side-effects is critical to system reliability.

* **Node State Machine:**
    Every node can only be in one of the following five states during its lifecycle:
    `PENDING` -> `RUNNING` -> `SUCCESS` | `FAILED` | `CANCELLED_DUE_TO_UPSTREAM`

* **Checkpointing & Idempotency Contract:**
    To support self-healing and prevent tools with side-effects (such as database writes or API charges) from being executed multiple times, **the engine does not rely on developers to manually implement idempotency**.
    1. **Persist on Success:** When a node transitions to `SUCCESS`, the engine immediately hashes or serializes its output and saves it as a checkpoint.
    2. **Bypass on Re-run:** When the engine receives a DAG containing existing nodes (such as a remedial DAG), it checks for checkpoints before dispatching tasks. If a successful checkpoint exists, the engine directly returns the checkpoint data and **will never** re-trigger the underlying Tool execution. This guarantees exactly-once execution.

## 3. Failure Taxonomy & Routing

When a node crashes in the `asyncio` event loop, system halting must be safe and predictable.

* **Tool Execution Failure / Timeout:**
    * *Definition:* The target API refuses connection, returns unexpected results, or exceeds the configured execution time.
    * *Engine Behavior (Halting Strategy):* The node is marked as `FAILED`. The engine's Dependency Resolver immediately scans the DAG and forcibly marks all downstream nodes (both direct and indirect dependencies) as `CANCELLED_DUE_TO_UPSTREAM`.
        **It is strictly forbidden to use `asyncio.Task.cancel()` to forcibly interrupt threads**, to avoid semaphore deadlocks. Nodes marked as Cancelled will be naturally skipped by the event loop.
    * *Parallel Isolation:* Parallel branches that are not blocked and do not depend on the failed node will continue to execute until completion.

* **Dependency Resolution Error:**
    * *Definition:* Before executing a node, the engine finds that the required `${node_id.output}` variable does not exist or has an invalid format.
    * *Engine Behavior:* Treated as a fatal error. The node is marked as `FAILED`, and the above downstream blocking mechanism is triggered.

## 4. Remedial DAG Contract

When a disaster occurs and the self-healing mechanism is triggered, the Planner generates a new remedial DAG based on the disaster log.

* *System Guarantee:* A remedial DAG does **not** start a brand new pipeline from scratch, but instead performs a **context merge** with the current state. The engine reads previous checkpoints and only reschedules and executes the nodes that were `FAILED` and the remaining subgraph originally marked as `CANCELLED_DUE_TO_UPSTREAM`.
